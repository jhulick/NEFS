So, you decided to do away with the varchar person_id, eh.

 

My preliminary comments on Event:

 

*           I like the idea of using Event_Org for waiting list.  That’s

innovative, man.

*           Your general description of the Event table refers to

Patient_Appt and Physician_Appt.  I think you mean Event_Patient and

Event_Physician, don’t you?

*           The fourth statement in the description has “The association of

this Event table with the Person table will result in two children

tables of Event: the Event_Physician and the Event_Physician tables”:

the first Event_Physician should be Event_Patient.

Both typos.  Will fix, altho I'm not sure this detailed description of the table and it's role will stay in the schema.  Might be moved into a separate document detailing all these data.

*           I would move checkin_id, checkin_time, checkout_id, and

checkout_time columns from Event_Physician to the main Event table

unless you have strong reasons for keeping them there.

These fields are not in the main Event table because that table stores information for all sorts of events, not just appointments.  Since these fields are relevant only to appointments and, therefore, the doctor side of the event, they're included in the Event_Physician table.  LMK if you guys think it's a good idea to move this into Event and, if so, why?

*           Need to introduce Appointment_Type.  This would store

user-defined appointment types.  Then tie Appointment_Type to Event.

This should probably be incorporated into the Event_Type table.  Thoughts?

*           An appointment type may specify other associated resources like

a nurse, an x-ray machine, an operating room, etc  so when making an

appointment using an appointment type, we’ll do the “explosion” thing,

making multiple Event records and associated Event_Physician records as

required.

*           And, BTW, need to have Event_Resource table for storing events

(or appointments) with non-human resources.

*           And, of course, a Resource_Item table for defining those

non-human resources.

Very cool!  Adds a whole new layer of functionality/complexity there.  Basically this takes the app into heavy machinery inventory and scheduling.  Do we need that right now or is it better to allow that to be added on later?

*           And speaking of explosion appointments, there ought to be

“master_event_id” or something.  That way when we cancel or reschedule

an appointment, we can replicate the change throughout the associated

appts.

Might be better to create an Event_Relationship table that'll store relationships between multiple events.  Just to keep the overall db design consistent (see Person/Org tables and transferring of hierarchical relationship info to the _Relationship tables).
